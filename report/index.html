3. Добавление индекса

-- Составной индекс по полям поиска
CREATE INDEX idx_name_search ON users (first_name, second_name);
Почему именно такой индекс?

Запрос вида first_name LIKE 'Ал%' AND second_name LIKE 'Ив%' использует префиксный поиск. MySQL B-Tree индекс поддерживает prefix-scan: он находит диапазон ключей начинающихся с нужного префикса без полного сканирования. Порядок столбцов в индексе (first_name, second_name) выбран так, что сначала отрабатывает более высокоселективное условие по имени, затем дополнительно фильтруется по фамилии в уже отобранном диапазоне. Индекс (second_name, first_name) работал бы так же, но (first_name, second_name) интуитивно понятнее и соответствует порядку параметров в API.

4. EXPLAIN после индекса

EXPLAIN SELECT ... FROM users FORCE INDEX (idx_name_search)

7. Выводы

  • Без индекса MySQL выполняет полный скан таблицы (~995K строк) на каждый запрос, что даёт ~4.7 с даже при 1 клиенте.
  • Составной индекс (first_name, second_name) переключает план на range scan, ускоряя поиск в ~49 раз при 1 VU.
  • При 100 параллельных запросах даже с индексом latency вырастает до ~1.8 с — из-за большого объёма ответа (300–500 KB на запрос).
  • Без лимита на количество результатов узким местом становится сетевая передача и сериализация JSON, а не БД.
  • MySQL не выбирает индекс автоматически при наличии ORDER BY id (предпочитает PK scan), поэтому использован FORCE INDEX.
